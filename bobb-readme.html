● Bobb — JSON Database with HTTP Access                                                                                                                                              
                                                                                                                                                                                              
  Bobb is a lightweight JSON database server for Go applications. It wraps bbolt — a proven embedded key-value store — and adds an HTTP layer, a query engine, and automatic secondary index  
  management on top of it.                                                                                                                                                                    
                                                                                                                                                                                              
  What it gives you

  - Simple data model. Records are JSON objects stored in named buckets. Each record is keyed by a string field of your choice (e.g. "id"). No schema definition required.                    
  - HTTP access. The server runs as a standalone process. Any number of client programs connect to it over HTTP using the provided Go client library, so multiple services can share one
  database safely.                                                                                                                                                                            
  - Query engine. Filter records with conditions on string fields (StartsWith, Contains, ContainsWord, Before, After, Matches, InList) and integer fields (Equals, LessThan, GreaterThan,
  InList). Combine conditions with AND or OR logic. Sort results by one or more fields, ascending or descending. Paginate with Limit/Top and NextKey.                                         
  - Secondary indexes. Declare an index once via IndexSettingRequest and bobb automatically maintains it on every write — including updates. Query a bucket through an index to read records
  in index order or restrict the scan to a key range.                                                                                                                                         
  - Joins. Pull fields from a related bucket into query results. Joins can run before or after the find step, so joined values can participate in filter conditions or be used only for
  sorting and display.                                                                                                                                                                        
  - Standard operations. Get by key, get all, key range reads, put (single or batch, multiple buckets in one request), delete, export to JSON, copy the live database to a file.
                                                                                                                                                                                              
  Good fit for    
                                                                                                                                                                                              
  - Go applications that need a persistent, queryable store without the overhead of a full SQL database or a separate database server to operate.                                             
  - Projects where multiple processes or services need to share data through a simple HTTP interface.
  - Workloads with moderate data volumes where the simplicity of JSON records and named buckets is a better match than a relational schema.                                                   
                                                                                                                                                                                              
  Not a good fit for                                                                                                                                                                          
                                                                                                                                                                                              
  - Very large datasets or high write throughput — bbolt is a single-writer database, so Bobb inherits that constraint.                                                                       
  - Applications that need transactions spanning multiple unrelated operations, complex aggregations, or full-text search.
  - Environments that require replication or high availability.                                                                                                                               
                                                                                                                                                                                              
  Getting started                                                                                                                                                                             
                                                                                                                                                                                              
  The server is a single Go binary configured with a small JSON settings file. Clients import the github.com/jayposs/bobb and github.com/jayposs/bobb/client packages, point bo.BaseURL at the
   server, and start making typed requests. The demo/ directory contains a complete working example that exercises every feature.
